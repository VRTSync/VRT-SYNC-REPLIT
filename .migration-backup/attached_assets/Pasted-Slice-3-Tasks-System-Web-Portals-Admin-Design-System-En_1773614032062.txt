Slice 3 — Tasks System (Web Portals) + Admin Design System Enforcement
Objective

Implement the Tasks experience for the new web portals (Contractor, HOA Admin, Property Manager) while strictly enforcing reuse of the existing admin portal design system and styling.

The goal is to ensure that the new web portals visually match the admin portal exactly and use the same CSS component classes, layout patterns, and interaction styles.

The Tasks system is the first operational module implemented in the web portal expansion.

Design System Rule (Critical)

All new portal pages must use the existing admin design system.

The following file is the single source of truth for styling:

server/public/admin/admin.css

This file defines:

color variables

typography

button styles

badges

layout helpers

form styles

modal styles

stat cards

list cards

No new design system should be created.

Brand Tokens (Already Defined)
--navy: #0C1D31
--teal: #25C1AC

Typography:

Inter

These values must remain the standard across all portals.

Required UI Component Reuse

All new pages must reuse the existing classes.

Buttons

Use existing classes:

.btn

.btn-secondary

.btn-danger

Do not create new button classes.

Badges / Chips

Use the existing badge system.

Examples:

.badge

priority chips

urgency chips

Dashboard Cards

Use existing dashboard card styles:

Forms

Use the admin form styling:

.form-group

All forms across portals must use these classes.

Modals

Reuse the modal system:

.modal-overlay

Existing dynamic modal pattern must be used.

via the existing router.

Tasks System Overview

The Tasks module powers the operational workflow for the platform.

Two task types exist:

Contract Tasks
HOA Requests

Both are stored in the same tasks table.

Tasks Page Implementation

Implement Tasks page for the following roles:

Contractor
HOA Admin
Property Manager

HOA Member does not need the full Tasks page yet.

Page Routes

Add routes for:

/web/contractor/tasks
/web/hoa/tasks
/web/pm/tasks

All routes must use the shared SPA router.

Example pattern:

Router.register('tasks', async function(container) {
   container.innerHTML = ...
});
API Endpoint

Use the existing endpoint:

GET /api/tasks

Server-side logic already scopes tasks correctly based on role.

Do not add client-side filtering logic.

Task Grouping Logic

Tasks must be grouped by window classification.

Rules:

Overdue
Active Window
Upcoming
No Window

Classification rules:

Overdue: today > windowEnd
Active: windowStart ≤ today ≤ windowEnd
Upcoming: today < windowStart
No Window: no window dates

Create a utility function:

classifyTask(task, today)
Task Card UI

Tasks should render as card-style list items, not tables.

Each card should include:

priority indicator

title

urgency chip

service window

address/location

request badge if applicable

Example card layout:

[Priority Dot] Task Title

Service Window
Location

Urgency Chip
Priority Colors

Map priority to color:

low = green
medium = orange
high = red
urgent = purple

Use badge classes for chips.
