SLICE MC1 — Map Creator Foundation: Role, Schema, Portal Shell (Mobile + DB + API + Admin)

Goal
Lay the structural foundation for the new Map Creator user type and its mobile portal, without building any map-creation UI yet. This slice (a) adds `map_creator` to `userRoleEnum`, (b) reconciles the `assetTypeEnum` with the types already referenced in `layerValidation.ts` and `assetSync.ts` by adding `master_valve`, `flow_meter`, `pump`, `quick_connect`, `isolation_valve` (and deliberately not adding the deprecated `qc_iso_valve`), (c) fixes the validation rule for `zone` to allow Point geometry (which is how zones are actually stored today), (d) creates an empty mobile route group `(mc-tabs)` parallel to `(tabs)` and `(hoa-tabs)` with placeholder Customers / Settings / Profile tabs, (e) updates the role-routing logic in `app/_layout.tsx` so `map_creator` users land in the new portal, (f) adds an `isMapCreatorRole()` server-side helper for permission checks, and (g) exposes `map_creator` in the admin user-edit form.

After this slice ships, an admin can promote any contractor to `map_creator`, that user can log in and lands in an empty Map Creator portal with three placeholder tabs, and the schema is ready for MC2 (Customer Selection) and beyond. No map UI, no GPS capture, no pin creation in this slice.

Scope (IN)

1) Database schema — enum updates and validation fixes
- Add `"map_creator"` to `userRoleEnum` in `lib/db/src/schema/schema.ts`.
- Add five values to `assetTypeEnum` in `lib/db/src/schema/schema.ts`:
  - `"master_valve"` — closes existing reference gap.
  - `"flow_meter"` — closes existing reference gap.
  - `"pump"` — new.
  - `"quick_connect"` — new, replaces half of `qc_iso_valve`.
  - `"isolation_valve"` — new, replaces other half of `qc_iso_valve`.
- Do **not** add `"qc_iso_valve"` to the enum. It is referenced in code today but never could insert because it was missing from the enum, so there is no production data to migrate.
- Generate the Drizzle migration: `pnpm --filter @workspace/db run migrate:generate`. Expect Drizzle to emit a migration like `0001_<auto-named>.sql` containing one `ALTER TYPE "user_role" ADD VALUE 'map_creator'` and five `ALTER TYPE "asset_type" ADD VALUE '...'` statements.
- **Important Postgres caveat**: `ALTER TYPE ... ADD VALUE` must run outside an explicit transaction. Drizzle's generated migration with `breakpoints: true` (already set in `lib/db/drizzle.config.ts`) handles this by separating each statement. Verify each `ALTER TYPE ... ADD VALUE` is on its own breakpoint (preceded by `--> statement-breakpoint`) before committing.

- Remove the `qc_iso_valve` entry from `EXPECTED_GEOMETRY` (no enum value, no rows possible).

6) Mobile — role-routing update in `artifacts/vrtsync-mobile/app/_layout.tsx`
- Inside `AuthNavigator`, replace the binary HOA/non-HOA logic with a three-way switch:
  ```ts
  const role = user?.role || '';
  const isHoa = role === 'hoa_admin' || role === 'hoa_member';
  const isMapCreator = role === 'map_creator';
  const correctStack: '(tabs)' | '(hoa-tabs)' | '(mc-tabs)' =
    isHoa ? '(hoa-tabs)' :
    isMapCreator ? '(mc-tabs)' :
    '(tabs)';
  const currentStack = segments[0];
  const inAuthGroup = currentStack === '(auth)';

  let didRedirect = false;

  if (!user && !inAuthGroup) {
    router.replace('/(auth)/login');
    didRedirect = true;
  } else if (user && inAuthGroup) {
    router.replace(`/${correctStack}` as any);
    didRedirect = true;
  } else if (user && currentStack && currentStack !== correctStack && !inAuthGroup) {
    router.replace(`/${correctStack}` as any);
    didRedirect = true;
  }
  ```
- Register the new stack alongside the others:
  ```tsx
  <Stack.Screen name="(mc-tabs)" options={{ headerShown: false }} />
  ```
- Behavior must be: any user with `role === 'map_creator'` who lands in `(tabs)` or `(hoa-tabs)` is redirected to `(mc-tabs)`; any non-map-creator who lands in `(mc-tabs)` is redirected to their correct stack.

B) Validation & sync code consistent with enum

Test plan
- Migration: applied on a copy of the dev DB; enum values present in psql; rolling back via `BEGIN; … ROLLBACK;` test against a scratch DB is acceptable since `ALTER TYPE ... ADD VALUE` cannot be rolled back within a transaction in Postgres.
- Manual end-to-end: covered in step 8 of Implementation order.
- Regression spot-check: a contractor and an HOA member each log in on the dev client and verify they land in their existing tab groups and see no UI changes.

Risk & rollback
- Risk: **low–medium**. Schema additions are additive (no column drops, no data migrations). The enum values cannot be removed once added in Postgres, so a true rollback of the migration would require recreating the enum — keep the values even on revert. Code rollback is straightforward (`git revert`).
- The riskiest piece is the role-routing rewrite in `app/_layout.tsx`; mitigated by manual login tests for all five existing roles plus map_creator in step 8.
- If the routing rewrite breaks an existing role, immediate fix is to revert just `app/_layout.tsx`; the schema and `(mc-tabs)` files can stay untouched and a follow-up fix can land.
